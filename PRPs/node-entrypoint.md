# PRP — Node Entrypoint, Config Loader, Service Wrapper, and Release Workflow

**Status:** pending approval
**Spec reference:** docs/specs/tetherdb.md §5.10, §5.13, §6 (steps 7 + 9)
**Implements:** `cmd/tetherdb/main.go`, `internal/config/`, `kardianos/service` integration, GitHub Actions release

---

## FEATURE

The `tetherdb` binary entrypoint: parses CLI flags and subcommands, loads and validates a TOML config file (with environment variable interpolation), wires `kardianos/service` for Windows Service / Linux systemd / macOS launchd, and produces release binaries (Linux amd64 + Windows amd64) via GitHub Actions on every push to `main`.

---

## OBJECTIVE

When a user runs `tetherdb` with no arguments, it prints help and exits 0. When run with `--config ./tetherdb.toml` (or the default config path), it loads and validates the config, calls `Probe()` on the configured connector to verify connectivity, then starts the node and blocks until a signal arrives. Service management subcommands (`install`, `uninstall`, `start`, `stop`, `run`) are routed through `kardianos/service`. On every push to `main`, GitHub Actions builds and uploads two release artifacts: `tetherdb-linux-amd64` and `tetherdb-windows-amd64.exe`.

## CLI INTERFACE

```
tetherdb [--config PATH] [--version] [SUBCOMMAND]

Subcommands:
  run         Start the node in the foreground (default when invoked by the service manager)
  install     Install tetherdb as a system service
  uninstall   Remove the system service
  start       Start the installed service
  stop        Stop the running service
  version     Print version and exit

Flags:
  --config PATH   Path to TOML config file (default: ./tetherdb.toml)
  --version       Print version and exit

No arguments: print help and exit 0.
```

The version string is injected at build time via `-ldflags "-X main.version=..."`. When not injected (local dev builds), it falls back to `"dev"`.

## CONFIG FILE STRUCTURE

Matches spec §5.10 exactly. The config package parses this TOML and exposes typed Go structs. Environment variable interpolation: any value of the form `${VAR_NAME}` is replaced with the value of `os.Getenv("VAR_NAME")` before the struct is populated. Interpolation applies to string fields only.

```toml
[node]
name     = "erp-node"
data_dir = "/var/lib/tetherdb"

[listen]
address  = "0.0.0.0:443"
tls_cert = "/etc/tetherdb/cert.pem"
tls_key  = "/etc/tetherdb/key.pem"

[management]
address = "127.0.0.1:8080"

[connector]
driver   = "sqlserver"
host     = "sqlserver.internal"
port     = 1433
database = "erp"
auth     = "sqlserver"
username = "${TETHERDB_SQL_USER}"
password = "${TETHERDB_SQL_PASS}"

[connector.publish]
tables = ["orders", "customers", "products"]

[[connections]]
name      = "primary-sink"
address   = "sink-node.internal:443"
subscribe = ["orders", "customers", "products"]
```

**Validation rules (checked at startup, before any connection):**

| Field | Rule |
|---|---|
| `node.name` | non-empty, alphanumeric + hyphens only |
| `node.data_dir` | non-empty string (existence not checked — created on first use) |
| `listen.address` | valid `host:port` format |
| `listen.tls_cert` / `tls_key` | non-empty; file existence checked at startup |
| `management.address` | valid `host:port`; must not be `0.0.0.0:*` |
| `connector.driver` | must be `"sqlserver"` (only supported driver in v1) |
| `connector.host` | non-empty |
| `connector.port` | 1–65535 (default 1433) |
| `connector.auth` | `"sqlserver"` or `"windows"` |
| `connector.username` | required when `auth = "sqlserver"` |
| `connector.publish.tables` | at least one entry |
| `connections[].name` | non-empty, unique across connections |
| `connections[].address` | valid `host:port` |
| `connections[].subscribe` | at least one entry; each must appear in `connector.publish.tables` |

Password is never logged. Management address must not bind to `0.0.0.0` — error if attempted.

## IMPLEMENTATION REQUIREMENTS

### Must Do

- Create `cmd/tetherdb/main.go` — entrypoint, flag parsing, subcommand dispatch
- Create `internal/config/config.go` — Config struct, `Load(path string) (*Config, error)`, env var interpolation, validation
- Register `kardianos/service` with the binary; route `install`/`uninstall`/`start`/`stop`/`run` through the service interface
- The service `Start()` method: loads config, calls `sqlserver.New()` + `Probe()`, logs startup line, then blocks on a `context.Context` that is cancelled on `Stop()`
- If `Probe()` fails at startup, log the error and return it — do not start the service
- Print help text (listing all subcommands and flags) when no arguments are provided
- Inject version at build time via `-ldflags "-X main.version=v0.1.0"`; fall back to `"dev"`
- Create `.github/workflows/release.yml` — triggers on push to `main`; builds `linux/amd64` and `windows/amd64` with `CGO_ENABLED=0`; strips debug info with `-ldflags="-s -w -X main.version=${VERSION}"`; uploads both binaries as workflow artifacts; version is derived from `git describe --tags --always --dirty`
- Every exported symbol must have a doc comment per `docs/CODE_STYLE.md`
- `CGO_ENABLED=0` at all times — no CGO anywhere

### Must NOT Do

- Do not implement the pipeline engine, transport, WebSocket, state layer, or management API — those are future PRPs
- Do not hardcode any host, port, credential, or path
- Do not log passwords or interpolated credential values
- Do not bind the management API listener in this PRP — just validate the address from config
- Do not panic on config errors — return errors to the caller
- Do not use `os.Exit` directly outside of `main()` — the service wrapper handles process lifecycle

## ERROR HANDLING REQUIREMENTS

| Situation | Behaviour |
|---|---|
| Config file not found | `Load()` returns descriptive error naming the path; node exits with non-zero code |
| Config file invalid TOML | `Load()` returns parse error with line number from TOML library |
| Validation failure | `Validate()` returns error naming the failing field and rule |
| `${VAR}` references undefined env var | interpolation leaves the value empty; validation catches required-field violations |
| `Probe()` fails at startup | log error, service `Start()` returns error, process exits non-zero |
| Unknown subcommand | print help and exit non-zero |

All errors wrapped with `fmt.Errorf("config: <context>: %w", err)` or `fmt.Errorf("main: <context>: %w", err)`.

## SECURITY CONSIDERATIONS

- Passwords and credentials must never appear in log output — log only `connector.username` and redact the password
- `management.address` must never be `0.0.0.0:*` — validate and error if attempted
- TLS cert and key file paths are validated for existence at startup — do not silently ignore missing files
- Env var interpolation must not expand arbitrary shell syntax — only `${VAR_NAME}` patterns (alphanumeric + underscore names); reject malformed patterns
